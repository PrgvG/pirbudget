# Готовность продукта PirBudget к запуску (бизнес-цель)

## Цель приложения

**Проблема:** Не контролирую финансы, не знаю, сколько и когда потребуется денег.

**Решение:** Приложение для планирования будущих трат и просмотра прошлых: ввод поступлений и обязательных платежей по группам, статистика за месяц.

---

## Текущее состояние

- **Есть:** авторизация (регистрация, логин, JWT), доменная модель в `shared` (категории, разовые/повторяющиеся расходы, поступления, recurrence), основной экран «Платежи и поступления», экран месяца со статистикой, историей и планом.
- **Нет:** части бэкенд-логики и/или API под отдельные сценарии (зависит от текущей реализации), а также доработок UX и онбординга.

---

## План по шагам

### 1. Группы платежей (категории)

- **Backend:** модель (Mongoose), CRUD API, привязка к `userId`.
- **Frontend:** экран «Группы» — список, создание/редактирование, порядок, цвет (иконка по желанию). Расходы выбирают группу из списка.

### 2. Поступления (доходы)

- **Backend:** модель поступлений (дата, сумма, источник, примечание), CRUD API, привязка к `userId`.
- **Frontend:** форма «Добавить поступление», список поступлений (таблица или карточки по датам).

### 3. Обязательные платежи (разовые и повторяющиеся)

- **Backend:** модели разовых и повторяющихся расходов + вхождения (occurrences). CRUD API, связь с группой и `userId`.
- **Frontend:** форма добавления платежа (группа, разовый/повторяющийся, сумма, дата или правило повторения), список обязательных платежей с редактированием/удалением.

### 4. Прошлые траты и поступления

- **Backend:** эндпоинты выборки операций за период (например, за месяц), фильтр по типу (доход/расход) и при необходимости по группе.
- **Frontend:** экран «История» — список операций за выбранный период (календарь, таблица или лента по датам). Показ и доходов, и расходов.

### 5. Будущие траты (план)

- **Backend:** расчёт/отдача «вхождений» повторяющихся платежей на выбранный период (или API для чтения/фильтрации occurrences).
- **Frontend:** экран «План» — календарь или список запланированных платежей (повторяющиеся + при желании разовые с датой в будущем). Цель: видеть «сколько и когда потребуется».

### 6. Статистика за месяц

- **Backend:** эндпоинт агрегации за месяц: сумма доходов, сумма расходов (всего и по группам), при необходимости баланс (доходы − расходы).
- **Frontend:** блок/экран «Статистика за месяц»: выбор месяца, итоги по доходам и расходам, разбивка по группам (список или простая диаграмма). Цель: один взгляд — картина месяца.

### 7. Привязка к пользователю и навигация

- Все сущности (группы, поступления, расходы) привязаны к залогиненному пользователю; API отдаёт только его данные.
- На фронте: навигация между разделами (Группы, Поступления, Платежи, История, План, Статистика) вместо текущего «списка пользователей».

### 8. Минимум UX для «запуска»

- Выбор месяца для истории и статистики (и при желании для плана).
- Понятные названия разделов и кнопки «Добавить поступление» / «Добавить платеж».
- Базовая валидация и сообщения об ошибках при сохранении.

---

## Итог

Доменная модель в `shared` уже под задачу; не хватает бэкенд-моделей и API под группы, поступления и расходы, привязки к пользователю и фронтовых экранов под ввод, прошлое, план и статистику. Выполнение пунктов 1–8 даёт готовность к первому запуску с точки зрения бизнеса.
